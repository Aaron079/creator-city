# ADR: Asset Remix Executor Architecture — Option B

**Date:** 2026-06-21
**Status:** ACCEPTED / DESIGN_FROZEN
**Deciders:** Founder (confirmed Option B)
**Supersedes:** Option A (direct RunPod adapter inside CC Vercel Function)

---

## Context

Phase 1A requires a GPU-backed background job pipeline to run SAM 2.1 subject extraction
on images uploaded to Aliyun OSS. Two architecture options were evaluated.

---

## Decision

**Option B: Independent Gateway Service on Railway Pro + Railway Redis**

```
Browser
  → CC Vercel (asset-transform route)           ← user auth, ownership verify, presigned URLs
    → Railway Gateway Service (FastAPI + HMAC)  ← RunPod orchestration, job state, output validation
      → RunPod Serverless (SAM 2.1 worker)      ← GPU inference, presigned PUT to OSS
    ← CC Internal Ingest API (Vercel + HMAC)    ← Asset creation, DB write
```

---

## Rejected Alternative

**Option A: Direct RunPod adapter inside CC Vercel Function**

Would have embedded RunPod polling, output validation, and Asset creation entirely within
the Vercel Function. Rejected for:

1. **Vercel execution limits.** Vercel Functions time out at 10–60s (plan-dependent).
   SAM 2.1 warm inference is 8–15s; cold start adds 15–30s. Polling loop inside a
   Vercel Function is structurally unreliable.
2. **Stateless constraint.** HMAC token as sole state carrier means no retry,
   no nonce replay protection, no TTL-based expiry of stale jobs.
3. **No background processing.** Vercel has no long-running worker model;
   output validation (HEAD + SHA-256) and ingest must happen outside Function scope.

---

## Consequences

### Positive
- Railway Pro containers run indefinitely; no Vercel timeout risk.
- Railway Redis provides job state, nonce replay protection, idempotency with TTLs.
- CC DB and OSS credentials stay inside CC Vercel; Gateway has no Prisma access.
- RunPod job IDs are never returned to the browser (ctid is the only client-visible ID).
- Output validation (HEAD + SHA-256 + PNG IHDR) happens server-side before any Asset is created.

### Negative / Constraints
- Additional infra: Railway Pro project + Railway Redis (managed).
- Two separate HMAC secrets: `ASSET_TRANSFORM_GATEWAY_SERVICE_TOKEN` (CC→Gateway)
  and `CC_INTERNAL_INGEST_TOKEN` (Gateway→CC).
- Requires SAM 2.1 SHA-256 verification before production deployment.
- Artifact gate (`_ARTIFACT_APPROVED = False` in gateway/app/main.py) must be set to `True`
  by Founder after artifact review.

---

## Key Design Constraints (binding)

1. `sourceMediaUrl` is NEVER accepted from the browser. CC Gateway resolves it from DB.
2. Presigned PUT URLs are generated by CC and forwarded to Gateway. NEVER returned to browser.
3. RunPod job ID (`rpjid`) is NEVER returned to the browser. `ctid` is the only client ID.
4. CC Internal Ingest API (`/api/internal/asset-transform/ingest`) creates Assets.
   Gateway Service has no Prisma/DB access.
5. HMAC scheme: `HMAC-SHA256(key, f"{timestamp}:{nonce}:{body_sha256_hex}")`.
   Timestamp window ±30s. Nonce replay tracked in Redis (10-min TTL).
6. `ASSET_TRANSFORM_INGEST_ENABLED` must be set in CC Vercel AND Founder must authorize
   before ingest route is active. Current state: `OUTPUT_INGESTION_IMPLEMENTATION_PENDING`.
7. Subject selection: score-based (5-factor weighted); NEVER select by largest area.
8. Source assets are NEVER modified.
