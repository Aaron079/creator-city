## 1. RunPod Serverless Native API (Reference)

All RunPod calls are made from the **Gateway Service on Railway Pro** (`runpod_client.py`). CC Gateway never calls RunPod directly — it calls the Gateway Service, which holds `RUNPOD_API_KEY` and `RUNPOD_ENDPOINT_ID`.

### Submit job (async)
```
POST https://api.runpod.io/v2/{RUNPOD_ENDPOINT_ID}/run
Authorization: Bearer {RUNPOD_API_KEY}
Content-Type: application/json

{
  "input": {
    "source_url":        "https://...aliyuncs.com/...?signature=...&expires=...",
    "output_key":        "assets/transforms/ct-xxxx/subject.png",
    "output_put_url":    "https://...aliyuncs.com/assets/transforms/ct-xxxx/subject.png?Signature=...&Expires=...",
    "mask_key":          "assets/transforms/ct-xxxx/mask.png",
    "mask_put_url":      "https://...aliyuncs.com/assets/transforms/ct-xxxx/mask.png?Signature=...&Expires=...",
    "transform_kind":    "remove-background",
    "ctid":              "ct-xxxxxxxxxxxxxxxx",
    "params":            { "mode": "auto", "feather_radius": 2 }
  }
}

→ 200 { "id": "runpod-job-id-abc123", "status": "IN_QUEUE" }
→ 400/422: invalid input (validation error in worker)
→ 429: rate limit (map to RUNPOD_UNAVAILABLE, retry with backoff)
→ 503: endpoint not ready (map to RUNPOD_UNAVAILABLE)
```

**Input field notes:**
- `source_url` — signed GET URL (5-minute expiry). Generated by `getSignedDownloadUrl()` in CC Gateway. Never from client.
- `output_put_url` and `mask_put_url` — presigned PUT URLs (15-minute expiry). Generated by fixed `getAliyunOssSignedUploadUrl()` in CC Gateway. Worker uses these to upload directly.
- `ctid` — creator transform ID (random prefix). Worker echoes it back in output. Used for key ownership verification.
- `params` — restricted allowlist; worker rejects any unknown fields.

### Poll job status
```
GET https://api.runpod.io/v2/{RUNPOD_ENDPOINT_ID}/status/{runpod_job_id}
Authorization: Bearer {RUNPOD_API_KEY}

→ {
    "id": "runpod-job-id-abc123",
    "status": "IN_QUEUE" | "IN_PROGRESS" | "COMPLETED" | "FAILED" | "CANCELLED" | "TIMED_OUT",
    "output": { ... },         ← present only when COMPLETED
    "error": "...",            ← present only when FAILED (worker error string)
    "executionTime": 12340,    ← ms from IN_PROGRESS to COMPLETED
    "delayTime": 3210          ← ms from submit to IN_PROGRESS (queue wait)
  }
```

### Cancel job
```
POST https://api.runpod.io/v2/{RUNPOD_ENDPOINT_ID}/cancel/{runpod_job_id}
Authorization: Bearer {RUNPOD_API_KEY}
→ 200 { "id": "...", "status": "CANCELLED" }
→ 404: job not found
→ 409: job already in terminal state
```

### Payload limits
- Input JSON: **10 MB max** — presigned URLs are ~1KB each; input is well within limit
- Output JSON (inside `output` field): **~10 MB max** — only metadata + keys, not binary
- Worker handler timeout: configured at endpoint level; Phase 1A = 120s hard limit
- Cold start: estimated 15–35s for 24GB GPU class with pre-cached Docker image

## 3. Status Mapping: RunPod → Creator City

| RunPod Status | Worker Output | Creator City Status | Notes |
|---|---|---|---|
| `IN_QUEUE` | — | `queued` | Waiting for GPU worker |
| `IN_PROGRESS` | — | `running` | Worker executing |
| `IN_PROGRESS` | progress: "uploading" | `uploading` | Worker PUTting to OSS (if progress supported) |
| `COMPLETED` | confidence=high | `ingesting` → `done` | CC Gateway runs ingestion pipeline |
| `COMPLETED` | error_code=NO_VALID_MASK | `failed` | CC maps error |
| `COMPLETED` | error_code=USER_SELECTION_REQUIRED | `failed` (Phase 1A) | `needs_user_selection` in Phase 1B |
| `COMPLETED` | error_code=ARTIFACT_HASH_MISMATCH | `failed` | Model integrity violation; alert ops |
| `FAILED` | error contains "OOM" | `failed` / `GPU_OOM` | |
| `FAILED` | error contains "MODEL_LOAD" | `failed` / `MODEL_LOAD_FAILED` | |
| `FAILED` | error contains "DECODE" | `failed` / `IMAGE_DECODE_FAILED` | |
| `FAILED` | error contains "UPLOAD" | `failed` / `OUTPUT_UPLOAD_FAILED` | Worker PUT failed |
| `TIMED_OUT` | — | `expired` / `JOB_EXPIRED` | Hard timeout hit |
| `CANCELLED` | — | `cancelled` / `JOB_CANCELLED` | |

## 6. Transform ID and Authorization Model

```
creatorTransformId (ctid):
  "ct-" + hex(sha256(userId + ":" + sourceNodeId + ":" + nonce)[0:16])
  Generated in CC Gateway. Never accepted from client body.

transformId (returned to client):
  HMAC token containing { ctid, rpjid, uid, pid, nid, op, iat, exp, nonce }
  See ASSET_REMIX_EXECUTOR_PHASE1A_DESIGN.md Section 5 for full token spec.

Polling authorization:
  On every GET /api/asset-transform?transformId={token}:
    1. verifyToken(token, sessionUserId) — checks HMAC + expiry + uid match
    2. If uid mismatch → 403 (cross-user attempt, log as security event)
    3. If expired → 410 with TRANSFORM_TOKEN_EXPIRED
    4. If tampered → 403 with TRANSFORM_TOKEN_INVALID
    5. Extract rpjid → call RunPod /status/{rpjid}
```

## 7. Idempotency

- `ctid` is deterministic from `(userId, sourceNodeId, nonce)`. Nonce is random per submit, so same node can have multiple active jobs.
- Duplicate poll: token verifies fine; returns current status. Idempotent by design.
- Duplicate submit (same nonce): CC checks `pendingAssetTransforms` in `CanvasWorkflow.metadataJson`; if active transform for `nodeId` exists with non-expired token → return existing token instead of new RunPod job.
- RunPod itself does not guarantee idempotency; dedup is handled in CC Gateway.

## 8. Security Requirements

| Secret | Location | Never in |
|---|---|---|
| `RUNPOD_API_KEY` | CC Vercel env (server-side only) | Client, logs, worker |
| `RUNPOD_ENDPOINT_ID` | CC Vercel env | Client body response |
| `ASSET_TRANSFORM_HMAC_SECRET` | CC Vercel env | Client, worker, RunPod input |
| Presigned PUT URLs | Generated server-side; sent to RunPod input | Never returned to frontend |
| Signed GET URLs | Generated server-side; sent to RunPod input | Never returned to frontend |
| RunPod job ID | Inside signed HMAC token | Never in plaintext response |

All tokens compared with `crypto.timingSafeEqual()`.

CC Gateway never accepts `outputAssetId`, `stableOutputMediaUrl`, `outputOwner`, or `ingestionStatus` from client request body. These are only set during server-side ingestion.
